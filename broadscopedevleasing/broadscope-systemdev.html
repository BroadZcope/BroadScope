Article 01

5 Questions to Ask Before
Building a Custom System

Most custom system projects fail not because of poor development, but because the decision to build was made before the problem was fully understood. These five questions will give you the clarity you need before you commit a single rand to development.

Why clarity before build is non-negotiable

A custom system is a significant investment — in money, time, and organisational change. Unlike buying off-the-shelf software, building custom means you own the decisions. Which means you also own the consequences of unclear thinking.

The companies that get the most value from custom systems are not necessarily the ones with the biggest budgets. They're the ones who did the thinking first.

Question 1 — What specific problem are we solving, and how do we know it's real?

This sounds obvious. It isn't. "We need a better system" is not a problem definition. "Our sales team is spending 4 hours a week manually reconciling data between our CRM and accounting software, causing an average of 3 invoicing errors per month" — that is a problem definition.

If you can't quantify the problem (in time, money, errors, or missed opportunities), you don't have enough clarity to build a solution. Get specific before you scope anything.

Test your problem statement: Can you explain the problem to someone outside the business in two sentences, and can they understand why it matters? If not, keep digging.

Question 2 — Have we exhausted off-the-shelf options?

Custom development is the right answer in specific circumstances — not by default. Before building, honestly evaluate whether existing tools can solve 80% of your problem. A custom system is justified when: existing tools can't handle your specific workflow, integration requirements are too complex for off-the-shelf solutions, you need a competitive moat that proprietary software can't provide, or the total cost of adapting your business to existing software exceeds the cost of building.

If none of those apply, you probably don't need to build custom yet.

Question 3 — What does this system need to connect to?

Integration points are where most systems fail. Before scoping a single feature, map every system your new solution needs to talk to: your CRM, your accounting software, your logistics platform, your communication tools. For each integration, determine: does the other system have an API, is that API well-documented, who owns and maintains the integration, and what happens if it breaks?

If you can't answer these questions before the build starts, your timeline and budget estimates are fiction.

Question 4 — Who owns this system after it's built?

Too many companies commission custom systems without planning for ownership. Systems need maintenance, updates, and iteration. Someone needs to be responsible for that. Before you build, decide: which internal team will own this system day-to-day, who approves changes and updates, who do you call when something breaks, and what's your relationship with the development team after launch?

Systems without clear ownership degrade. The investment you make in building depreciates without a plan for long-term stewardship.

Question 5 — What does success look like 12 months after launch?

Define your success metric before you build — not after. Typical metrics include time saved per week (quantified), error rate reduction, revenue attributable to the system, reduction in manual processes, or cost savings versus the previous approach. If you can't define what success looks like in advance, you won't know if you've achieved it — and you won't be able to justify further investment.

Article 02

How We Build Custom Systems
Without Integration Nightmares

Integration failures are the number one cause of system project delays. Data doesn't flow. APIs break. Two systems that were "supposed to connect" turn out to require months of custom work. Here's how BroadScope designs integrations upfront — so they work reliably, not occasionally.

Why integrations are where projects die

Most development teams build the core system first and "handle integrations later." This is backwards. Integrations define the architecture. The way data needs to flow between systems determines the database structure, the API design, the error handling logic, and the performance requirements. Building without mapping integrations first is like building a city without planning the roads.

We've seen projects double in timeline because an integration that was assumed to be simple turned out to require a complete rebuild of the core data model. This is avoidable — if you do the work upfront.

Phase 1 — Integration mapping before any code

Before we scope a single feature, we produce a full integration map. Every system that needs to connect to the solution is identified, along with the direction of data flow (one-way vs. bidirectional), the frequency of sync (real-time, scheduled, event-triggered), the API availability and quality of each connected system, the data transformation requirements, and the failure and fallback logic.

This map is reviewed by both our technical team and the client. If any integration is unclear or undocumented, that risk is surfaced before development starts — not discovered at week eight.

The rule we never break: Every integration point must have a documented failure mode before development begins. "It should just work" is not acceptable. "If the CRM API is unavailable, the system queues the update and retries every 15 minutes with an alert at 1 hour" — that is acceptable.

Phase 2 — Build integrations first, not last

We build integration layers before the UI. This means the data pipelines, transformation logic, and API connections are tested and stable before anything is built on top of them. Teams that build the UI first and integrations last are always forced to retrofit — and retrofitting integrations into an existing data model is expensive, fragile, and often requires rebuilding parts of the core system.

  • 1
    Integration layer built and tested first — data flows are confirmed before any feature development
  • 2
    Core system built on stable foundation — no retrofitting, no surprises
  • 3
    UI layer last — built on confirmed, stable data models

Phase 3 — Testing that actually catches integration failures

Integration failures are most commonly discovered in production — when real data, real load, and real edge cases expose gaps that test environments didn't catch. Our testing approach includes: staging environments that replicate production data volumes, edge case testing for every integration (what happens when the API returns unexpected data, when a sync fails, when data is missing), load testing for real-world usage scenarios, and manual testing by someone who wasn't involved in building the feature.

Integration testing is not optional. It is the difference between a system that works reliably and one that works until it doesn't.

What happens after launch

Integrations don't stay stable indefinitely. APIs get updated. External systems change their data models. Authentication methods expire. We build monitoring and alerting into every integration we deliver — so when something changes in a connected system, you know before your users do. Post-launch support includes: integration health monitoring, alerts for failures or unexpected data, and a documented process for handling breaking changes in connected systems.
