Article 01

How to Scope an MVP
Without Feature Creep

Most software projects don't fail because of bad code. They fail because nobody agreed on what "done" meant before the build started. Here's how to define your MVP with precision — so you launch fast and iterate with purpose.

Why feature creep kills MVPs

Feature creep doesn't start with bad intentions. It starts with a reasonable question: "While we're building this, shouldn't we also add…?" Each addition feels small. Collectively, they double your timeline, triple your budget, and guarantee you'll ship something that nobody has tested with real users.

The goal of an MVP is not to build something impressive. It is to answer one question as cheaply as possible: Will someone pay for this? Everything else is hypothesis.

The core principle: An MVP is not a lite version of your product. It is the minimum surface area required to validate your most important assumption.

The 3-question MVP framework

Before scoping a single feature, answer these three questions in writing. If you can't answer them clearly, you're not ready to build.

  • 1
    What is the single problem we are solving? Not a list. One problem. For one type of person. If your answer contains the word "and", cut it in half.
  • 2
    What is the minimum action a user needs to take to get value? Map it as a step-by-step flow. Every feature that doesn't appear in that flow is out of scope for v1.
  • 3
    How will we know if this is working? Define a success metric before you build — not after. "Users are signing up" is not a metric. "40% of users complete onboarding and return within 7 days" is.

How to classify every feature request

When a feature is proposed, run it through this filter before adding it to scope:

QuestionIf YesIf No
Does the MVP fail without this feature?Include itMove to v2 backlog
Can this be faked manually for now?Fake it (don't build)Evaluate further
Does it serve the core user flow?Consider itCut it
Can it be added after launch without rebuilding?Defer itPrioritise if core

The "fake it first" rule

Before building any backend logic, dashboards, or admin panels — ask whether it can be faked manually. Zapier instead of a custom integration. A spreadsheet instead of a database. A manual email instead of an automated sequence.

This isn't cutting corners. This is smart validation. You learn faster, spend less, and only build automation for processes you've confirmed are worth automating. Dropbox famously validated their entire product with a video before writing a single line of code.

Locking scope before development starts

Once you've answered the three questions and filtered your feature list, create a one-page scope document that includes: the core user flow (step by step), a list of what's explicitly in v1, a list of what's explicitly deferred to v2, and the success metric you'll measure at launch.

This document becomes your single source of truth. Every scope change request gets evaluated against it. Not against enthusiasm, not against "it would be nice to have" — against the validated MVP definition you agreed to.

How We Scope Custom Software
Without Scope Creep

Scope creep is the most common reason software projects go over budget and under-deliver. Here's the exact process BroadScope uses to define, lock, and protect the scope of every custom build — so you always know what you're getting and when you're getting it.

Why most scoping processes fail

Most development agencies scope projects in a single discovery call, write a proposal, and start building. The problem is that clients don't know what they want until they see what they don't want. Without a structured process, requirements evolve mid-build — and every change costs time, money, and momentum.

Our scoping process is designed to surface ambiguity before it becomes a problem. We ask uncomfortable questions early so there are no surprises later.

Phase 1 — Problem mapping (before we talk about features)

We start every project by mapping the problem, not the solution. This means understanding the current workflow (manual or broken), where the bottlenecks and failure points are, what a successful outcome looks like in measurable terms, and who the actual end users are (not just the person paying for it).

This phase typically takes one focused session. By the end, we have a written problem statement that both parties sign off on. This is not the spec — it's the constraint the spec must satisfy.

Our rule: We don't write a single line of code until the problem statement is locked. Features are solutions to problems. If the problem isn't defined, no feature is justified.

Phase 2 — User flow before UI

After the problem is mapped, we build the user flow as a plain-text or simple diagram — not wireframes, not mockups. Just: what does the user do, in what order, to get the outcome they need?

This step catches 80% of scope issues. We consistently find features that seemed important but aren't in the flow, and flows that reveal missing features nobody mentioned. User flow review is where we eliminate feature creep before it enters the backlog.

  • 1
    Map the happy path — the exact steps a user takes to complete the core action
  • 2
    Map the edge cases — what happens when something goes wrong or the user takes an unexpected action
  • 3
    Identify integration points — where does this system touch other systems? These are always where timelines slip.

Phase 3 — The locked scope document

Once the user flow is confirmed, we produce a locked scope document. This is not a proposal or an estimate — it is an agreement. It contains the full feature list with acceptance criteria for each item, explicit exclusions (what is not in scope and why), integration requirements and dependencies, and a phased delivery plan with milestones.

Both parties sign it. Any change request after sign-off goes through a formal change order process — not a Slack message. This protects both the client and the build team.

How we handle change requests mid-build

Change requests are inevitable. The question is whether they're controlled or chaotic. Our process: every change request is logged, assessed for impact on timeline and budget, and presented back to the client with options. Option A: add it now with a formal extension. Option B: defer it to v2. Option C: replace an existing feature.

What we never do is absorb changes silently. Silent scope expansion is how projects fail. Transparency about tradeoffs is how they succeed.
